Core Frame
What Big Idea 3 Is Really About

Big Idea 3 is the programming heart of AP CSP. It covers how programs are built: storing data in variables, controlling flow with conditionals and loops, grouping data in lists, and packaging logic into reusable procedures.

Beyond syntax, it also asks deeper questions: what makes an algorithm correct? How do we measure efficiency? And are some problems simply unsolvable?

Exam Focus
Be ready to trace pseudocode by hand — follow a loop or conditional step by step, track variable values, and predict output. The exam tests understanding, not memorization.

Common Mistakes

Avoid These

  • Confusing x ← x + 1 (update x) with x = x + 1 math — in code, assignment replaces the old value.
  • Thinking lists start at index 0 — AP CSP pseudocode lists start at index 1.
  • Using a binary search on an unsorted list — it only works when data is sorted.
  • Confusing REPEAT n TIMES (fixed count) with REPEAT UNTIL (condition-based) — a REPEAT UNTIL loop with a never-true condition runs forever.
  • Assuming more efficient always means faster — efficiency depends on input size and the rate of growth.
Study Strategy

Best Order To Review

  • Start with variables, assignment, and expressions — foundation for everything else.
  • Master sequencing → selection (IF/ELSE) → iteration (REPEAT loops, FOR EACH).
  • Practice tracing pseudocode by hand on paper before looking at the answer.
  • Study lists: indexing, INSERT, APPEND, REMOVE, LENGTH, and traversal patterns.
  • Nail procedures and parameters — abstraction is the most conceptual part of BI 3.
  • Finish with algorithm efficiency, linear vs. binary search, and decidability.

Major Buckets
01 · Variables & Data

Storing and Updating Values

A variable is a named storage location whose value can change as a program runs. In AP CSP pseudocode, the assignment operator is ←. The right side is evaluated first, then stored into the variable on the left.

Assignment
x ← 5 stores 5 in x. x ← x + 1 reads the current value of x, adds 1, and stores the result back. The old value is replaced.
Data Types
Variables can hold numbers (integers and decimals), strings (text in quotes), and Booleans (true/false). The type determines what operations are valid.
String Concatenation
Joining strings together. Example: "Hello" + " " + "World" → "Hello World". Strings and numbers are different — "5" is not the same as 5.
Common Error
Reading x ← x + 1 as an equation — it isn't. It's an instruction: "calculate x + 1, then store it in x." The old x is gone.

Tracing Variable Changes

x ← 3 y ← x + 2 // y = 5 x ← x * y // x = 3 * 5 = 15 DISPLAY(x) // outputs 15
02 · Arithmetic & Boolean Expressions

Evaluating Expressions

Arithmetic expressions combine variables, values, and operators (+, −, *, /) into a number. The MOD operator returns the remainder of division (e.g., 17 MOD 5 = 2). Boolean expressions evaluate to true or false and are the foundation of all decision-making in programs.

MOD Operator
n MOD 2 = 0 means n is even. n MOD 2 = 1 means n is odd. MOD is used constantly to detect divisibility and wrap around values.
Comparison Operators
= (equal), ≠ (not equal), < (less than), > (greater than), ≤, ≥. These return true or false and are used inside IF conditions and REPEAT UNTIL loops.
AND
Both conditions must be true for the whole expression to be true. If either is false, the result is false.
OR
At least one condition must be true. Only false when both are false.
NOT
Flips the Boolean value. NOT true = false; NOT false = true.
Tip
On the exam, mentally substitute true/false for conditions and trace AND/OR/NOT outcomes step by step. Don't guess from intuition.

Truth Table Quick Reference

A=true, B=true → A AND B = true, A OR B = true A=true, B=false → A AND B = false, A OR B = true A=false, B=true → A AND B = false, A OR B = true A=false, B=false → A AND B = false, A OR B = false NOT true = false, NOT false = true
03 · Selection (Conditionals)

Making Decisions in Code

Selection lets a program choose which block of code to run based on a Boolean condition. Without selection, programs can only run in a straight line — no choices. The fundamental structure is IF / ELSE IF / ELSE.

IF Statement
Runs its block only when the condition is true. If false, the block is skipped entirely.
IF / ELSE
Runs the IF block if true, the ELSE block if false. Exactly one of the two blocks executes.
Nested Conditionals
An IF inside another IF. Allows more complex decision trees. Pay attention to which ELSE belongs to which IF when tracing nested code.
AP Move
When tracing conditionals, evaluate the condition first — don't assume which branch runs. Substitute actual variable values into the Boolean expression before deciding.

AP CSP Pseudocode: IF / ELSE

score ← 85 IF score >= 90 { DISPLAY("A") } ELSE IF score >= 80 { DISPLAY("B") // This branch runs: 85 >= 80 is true } ELSE { DISPLAY("C or below") }
04 · Iteration (Loops)

Repeating Instructions

Iteration allows a program to execute the same block of code multiple times without rewriting it. There are three main loop types in AP CSP pseudocode: REPEAT n TIMES, REPEAT UNTIL, and FOR EACH.

REPEAT n TIMES
Runs the block exactly n times. Use when you know the count in advance.
REPEAT UNTIL (condition)
Runs until the condition becomes true. The body runs at least once, then checks. Warning: if the condition can never be true, this creates an infinite loop.
FOR EACH item IN list
Assigns each element to a temporary variable one at a time and runs the block. Ideal for processing every element in a list.
Infinite Loop
When the exit condition is never met, the loop runs forever. Common cause: the variable controlling the condition is never updated inside the loop.

REPEAT UNTIL: Find Sum

sum ← 0 i ← 1 REPEAT UNTIL i > 5 { sum ← sum + i i ← i + 1 } DISPLAY(sum) // outputs 15 (1+2+3+4+5)

FOR EACH: Find Max in List

nums ← [3, 7, 2, 9, 4] maxVal ← nums[1] FOR EACH n IN nums { IF n > maxVal { maxVal ← n } } DISPLAY(maxVal) // outputs 9
05 · Lists (Arrays)

Storing Multiple Values

A list stores an ordered collection of values in a single variable. Each value is called an element, accessed by its index (position). In AP CSP pseudocode, indices start at 1, not 0.

Index Access
myList[1] is the first element. myList[LENGTH(myList)] is the last. Accessing an index that doesn't exist causes an error.
APPEND
APPEND(myList, value) adds value to the end of the list. List length increases by 1.
INSERT
INSERT(myList, i, value) inserts value at index i, shifting all elements at and after i one position to the right.
REMOVE
REMOVE(myList, i) deletes the element at index i, shifting remaining elements left. List length decreases by 1.
LENGTH
LENGTH(myList) returns the number of elements in the list. Use this in loops to avoid hard-coding the count.
Traversal
Iterating over every element in a list. A full traversal visits all elements; a partial traversal visits a subset. FOR EACH is the cleanest way to traverse.

Computing Average of a List

grades ← [88, 92, 76, 95, 84] total ← 0 FOR EACH g IN grades { total ← total + g } avg ← total / LENGTH(grades) DISPLAY(avg) // outputs 87
06 · Procedures & Abstraction

Packaging and Reusing Logic

A procedure (also called a function) is a named block of code that runs only when called. Procedures allow programmers to package logic once and reuse it throughout a program without rewriting it.

Procedural abstraction means a caller only needs to know: the procedure's name, what parameters it takes, and what it returns — not how it works internally. This hides complexity and reduces errors.

Parameters
Variables that receive values when a procedure is called (the arguments). They allow the same procedure to work with different inputs each time it's called.
Return Statement
Sends a value back to the calling code and immediately ends the procedure. Not all procedures return a value — some just perform an action.
Calling a Procedure
When called, the program pauses, executes the procedure, then continues from where it left off. Arguments are passed to parameters in order.
Built-in Procedures
Pre-written, tested procedures included with the language. Examples: DISPLAY, LENGTH, APPEND, RANDOM. You call them without writing the code yourself.
Libraries & APIs
A library is a collection of procedures. An API (Application Programming Interface) describes what procedures are available and how to call them.
Modularity
Splitting a program into separate, independent procedures — each responsible for one thing. Makes programs easier to test, debug, and modify.

Procedure with Parameter and Return

PROCEDURE square(n) { RETURN n * n } result ← square(7) DISPLAY(result) // outputs 49

Procedure that Acts (no return)

PROCEDURE greet(name) { DISPLAY("Hello, " + name + "!") } greet("Alex") // outputs: Hello, Alex!
07 · Algorithms

Step-by-Step Problem Solving

An algorithm is a finite, precise sequence of instructions that solves a problem or computes a result. Every program implements one or more algorithms. A correct algorithm must: terminate (not run forever), produce the correct output for all valid inputs, and handle edge cases.

Sequencing
Instructions run in order, one after another. The sequence matters — swapping steps can change or break the result.
Selection
A decision point (IF/ELSE) that takes one of two or more paths based on a condition.
Iteration
A loop that repeats steps. Combined with sequencing and selection, these three building blocks can express any computable algorithm.
Multiple Correct Algorithms
Many different algorithms can solve the same problem correctly. They may differ in speed, clarity, or resource usage. Correctness and efficiency are separate concerns.
Logic Error
The algorithm runs without crashing but produces the wrong result. Hard to detect because the program doesn't signal a problem.
Debugging
Techniques: hand tracing (simulating the algorithm manually), adding DISPLAY statements, using test cases, and visualizing data flow.

Common Algorithm Patterns

// Find minimum value in a list list ← [8, 3, 6, 1, 9] minVal ← list[1] FOR EACH n IN list { IF n < minVal { minVal ← n } } DISPLAY(minVal) // outputs 1
08 · Searching Algorithms

Linear Search vs. Binary Search

Searching means finding a specific value (the target) in a dataset, or confirming it isn't there. The two key algorithms are linear search and binary search — they differ enormously in efficiency.

Linear Search
Checks each element one by one from the start. Works on any list (sorted or not). Worst case: must check every element. Good when: list is small or unsorted.
Binary Search
Repeatedly splits a sorted list in half. Compare target to the middle element — if target is smaller, search the left half; if larger, search the right half. Requires: list must be SORTED first.
Efficiency Gap
Linear search: up to n comparisons for a list of n items. Binary search: at most log₂(n) comparisons. For 1,000,000 items — linear needs up to 1,000,000 checks; binary needs at most 20.
Critical Rule
Binary search ONLY works on a sorted list. Using it on an unsorted list produces incorrect results — it will "find" nothing even if the item exists.

Binary Search Trace: Find 7 in [1, 3, 5, 7, 9, 11, 13]

List: [1, 3, 5, 7, 9, 11, 13] (7 elements) Step 1: middle = index 4 → value 7 target (7) = middle (7) → FOUND at index 4

Binary Search Trace: Find 10 in [2, 5, 8, 12, 16, 23]

Step 1: middle = index 3 → value 8 | 10 > 8 → search right half Step 2: right half: [12, 16, 23] | middle = 16 | 10 < 16 → search left Step 3: left: [12] | 12 ≠ 10 → NOT FOUND
09 · Algorithm Efficiency

Reasonable vs. Unreasonable Time

Efficiency is about how an algorithm's runtime grows as the input size grows. AP CSP distinguishes between reasonable time (polynomial growth — n, n², n³) and unreasonable time (exponential growth — 2ⁿ). For large inputs, unreasonable algorithms become practically unusable.

Reasonable Time
Runtime grows as a polynomial of input size (n, n², n³). Even n² can be acceptable for moderate n. As input grows, time grows at a manageable rate.
Unreasonable Time
Runtime grows exponentially (2ⁿ, n!). Even for n = 50, 2⁵⁰ is over a quadrillion operations — impossible with any computer. These algorithms don't scale.
Heuristics
When an exact solution is unreasonably slow, a heuristic finds a "good enough" answer quickly. Not guaranteed optimal, but practical. Used for problems like the traveling salesman.
Parallel Computing
Running tasks simultaneously on multiple processors. Can speed up solutions when tasks are independent. Does NOT make exponential problems polynomial — it just reduces the constant.

Growth Rate Comparison

Input n = 10: n = 10 operations n² = 100 operations 2ⁿ = 1,024 operations Input n = 50: n = 50 operations n² = 2,500 operations 2ⁿ = 1,125,899,906,842,624 operations (impossible!)
10 · Undecidability & Limits

Problems Algorithms Cannot Solve

Not every problem has an algorithmic solution. A problem is decidable if an algorithm can always answer yes or no for every possible input. A problem is undecidable if no algorithm can correctly answer yes or no for all inputs — proven by logic, not just a matter of not yet finding the right algorithm.

The Halting Problem
Given a program and its input, will it halt (finish), or run forever? Proven to be undecidable — no algorithm can correctly answer this for every program/input combination. This is the canonical undecidable problem on the AP exam.
Undecidable ≠ Unsolvable for All Cases
An undecidable problem may be solvable for specific inputs — it just can't be solved for ALL inputs by any single algorithm.
Implications
These limits exist for all computers — no increase in computing power changes them. They're mathematical limits, not engineering limits. Even a theoretically perfect computer cannot solve undecidable problems.
AP Move
When asked about undecidability: state that no algorithm exists that always gives the correct answer for all inputs, and cite the halting problem as the key example. Never say "we just haven't found it yet."

Exam Prep

AP Exam Strategies

High-yield patterns and common traps for Big Idea 3

Variables & Expressions

Tracing Variable State

  • Always evaluate the right side of ← before storing into the variable.
  • x ← x + 1 adds 1 to x and saves it; x's old value is gone.
  • MOD returns the remainder: 17 MOD 5 = 2; use to test even/odd (n MOD 2).
  • Trace code in a table: one column per variable, one row per step.
Boolean Logic

AND / OR / NOT Rules

  • AND: both must be true. Any false → result is false.
  • OR: at least one must be true. Both false → result is false.
  • NOT: flips the Boolean. NOT (x > 5) is the same as x ≤ 5.
  • Substitute concrete values before evaluating compound expressions.
Loops

REPEAT vs. FOR EACH

  • REPEAT n TIMES → use when count is known ahead of time.
  • REPEAT UNTIL (condition) → use when you stop on a condition; runs at least once.
  • FOR EACH item IN list → cleanest way to process every element.
  • Infinite loop = condition never becomes true → update variables inside the loop!
Lists

Index Rules & Operations

  • AP CSP lists are 1-indexed: first element is at index 1, not 0.
  • APPEND adds to the end; INSERT shifts elements right from the target index.
  • REMOVE shifts elements left; list shrinks by 1.
  • LENGTH(list) is the safest way to reference the last element: list[LENGTH(list)].
  • Accessing an out-of-range index causes a run-time error.
Procedures

Abstraction & Parameters

  • To use a procedure, you only need name + parameters + return type — not the internal code.
  • Arguments (at the call) map to parameters (in the definition) in order.
  • RETURN sends a value back and immediately ends the procedure.
  • Modularity: one procedure, one job. Easier to test and debug independently.
Search & Efficiency

Linear vs. Binary

  • Linear search: O(n) worst case — works on any list.
  • Binary search: O(log n) — only works on a sorted list.
  • Undecidable problems (like the Halting Problem) cannot be solved by any algorithm for all inputs — this is a mathematical limit, not a hardware limit.
  • Reasonable time = polynomial (n, n²). Unreasonable = exponential (2ⁿ).
